# Troubleshooting

## OSAC Fulfillment-Service: Dirty Database Migration

**Symptom:** The OSAC gRPC server crashes in a loop with:
```
Dirty database version 69. Fix and force version.
```

**Root cause:** Migration 69 (`69_add_project_column.up.sql`) calls
`uuidv7()`, which is a **PostgreSQL 18 built-in function**. If you run
OSAC against PostgreSQL 16 or 17, this function doesn't exist and the
migration crashes. The go-migrate library marks `schema_migrations` as
`dirty=true` before running each migration, so the crash leaves the DB
in a dirty state that blocks all subsequent starts.

The OSAC team uses PG 18 (via `quay.io/sclorg/postgresql-18-c10s`) in
their own infrastructure, so they never hit this.

**When it happens:**
- Running OSAC fulfillment-service against PostgreSQL < 18
- Migration 69 calls `uuidv7()` → PG throws "function does not exist"
- go-migrate has already set `dirty=true` on `schema_migrations`
- Every pod restart sees `dirty=true` and exits immediately

**Fix (manual / one-time):**
```bash
# Option 1: Reset the dirty flag (preserves existing tables)
kubectl exec -n osac statefulset/osac-db -- \
  psql -U osacuser -d osacdb -c "UPDATE schema_migrations SET dirty = false;"

# Option 2: Drop everything and let migrations re-run from scratch
kubectl exec -n osac statefulset/osac-db -- \
  psql -U osacuser -d osacdb -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"

# Then restart gRPC to pick up the clean state
kubectl rollout restart deployment/osac-grpc -n osac
```

**Fix (CI / automation):**
Use an init container that waits for PostgreSQL AND resets the schema
before the gRPC container starts. See `integration-test/deploy-osac.sh`
for the pattern.

**Prevention:**
- Always ensure PostgreSQL is fully ready before deploying gRPC
- Use readiness probes on the PostgreSQL pod
- On CRC/OpenShift: CloudNativePG handles migrations correctly and
  avoids this issue entirely (see `docs/dev/crc-osac-deployment.md`)

**Related:**
- go-migrate issue: https://github.com/golang-migrate/migrate/issues/283
- Our CRC deployment used CloudNativePG which handles this correctly
- The k3s full-stack script ([`snippets/k3s-full-stack.sh`](../../snippets/k3s-full-stack.sh))
  uses an init container to work around this

## Consumer 401: Token Not Valid (OSAC Event Stream)

**Symptom:** `cost-event-consumer` loops with:
```
WARN event stream disconnected, reconnecting error="watch returned status 401: {\"error\":{\"code\":16,\"message\":\"token is not valid\"}}"
```

**Root cause (1): Dummy token in secret.** The initial deploy puts
`dummy-token-for-testing` in the `cost-consumer-secrets` secret. This is
never valid — generate a real token (see below).

**Root cause (2): Token expiry.** Tokens generated by `gen_token.py` expire
after 24 h. Re-run and update the secret.

**Root cause (3): Wrong signing key.** The `osac-oidc` deployment originally
mounted `osac-grpc-tls` to serve its HTTPS endpoint. That cert's SANs only
cover `osac-grpc.osac.svc.*`, so the OSAC gRPC server's TLS handshake to
`https://osac-oidc.osac.svc:8013` fails:
```
x509: certificate is valid for osac-grpc.osac.svc.cluster.local, not osac-oidc.osac.svc
```
The gRPC server can't fetch the JWKS → every token is rejected. This was
fixed by issuing a dedicated `osac-oidc-tls` certificate (see fix below).

**Fix — generate a valid token:**
```bash
# Extract the OIDC signing key (must match what osac-oidc pod serves in JWKS)
eval $(crc oc-env)
oc get secret osac-oidc-tls -n osac -o jsonpath='{.data.tls\.key}' | base64 -d > /tmp/osac-oidc.key

# Generate JWT (requires: pip install cryptography pyjwt)
python3 inventory-watcher/scripts/gen_token.py   # sets FULFILLMENT_SERVICE_DIR or edit key_path

# Or inline with the correct key:
TOKEN=$(python3 - <<'EOF'
import json, hashlib, base64, datetime
from cryptography.hazmat.primitives.serialization import load_pem_private_key
import jwt
with open("/tmp/osac-oidc.key", "rb") as f:
    pk = load_pem_private_key(f.read(), password=None)
pub = pk.public_key().public_numbers()
def b64url(n):
    l = (n.bit_length()+7)//8
    return base64.urlsafe_b64encode(n.to_bytes(l,"big")).rstrip(b"=").decode()
jwk = {"e": b64url(pub.e), "kty": "RSA", "n": b64url(pub.n)}
tp = json.dumps(jwk, separators=(",",":"), sort_keys=True)
kid = base64.urlsafe_b64encode(hashlib.sha256(tp.encode()).digest()).rstrip(b"=").decode()
now = datetime.datetime.now(datetime.timezone.utc)
print(jwt.encode(
    {"iss": "https://osac-oidc.osac.svc:8013", "sub": "admin",
     "preferred_username": "admin", "groups": ["admins"],
     "iat": now, "exp": now + datetime.timedelta(hours=24)},
    pk, algorithm="RS256", headers={"kid": kid}))
EOF
)

kubectl create secret generic cost-consumer-secrets -n cost-mgmt \
  --from-literal=osac-token="$TOKEN" --dry-run=client -o yaml | kubectl apply -f -
kubectl delete pod -n cost-mgmt -l app=cost-event-consumer
```

**Fix — wrong SAN on OIDC cert (one-time, already applied on CRC):**

The `osac-oidc` pod must serve TLS with a cert that covers `osac-oidc.osac.svc`.
Issue a dedicated cert and patch the deployment:

```bash
# 1. Issue certificate with correct SANs
oc apply -f - <<'EOF'
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: osac-oidc-tls
  namespace: osac
spec:
  secretName: osac-oidc-tls
  issuerRef:
    name: osac-ca
    kind: ClusterIssuer
  dnsNames:
    - osac-oidc
    - osac-oidc.osac
    - osac-oidc.osac.svc
    - osac-oidc.osac.svc.cluster.local
EOF

# 2. Patch the OIDC deployment to mount the new secret
oc patch deployment osac-oidc -n osac --type=json -p='[
  {"op": "replace", "path": "/spec/template/spec/volumes/1/secret/secretName",
   "value": "osac-oidc-tls"}
]'
```

**Diagnosis tip:** Check the gRPC server logs — they show the actual TLS/JWKS
error, while the consumer only shows the downstream 401:
```bash
oc logs -n osac deploy/osac-grpc --tail=20
```

**Related:** `crc-full-deployment.md` → "Generate OSAC Token" section (the
`gen_token.py` script used `localhost:8013` as issuer — must be
`osac-oidc.osac.svc:8013` for in-cluster deployments).
